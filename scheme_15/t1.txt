Так, тут проблема не у тобі — проблема в самій моделі даних і в тому, як зараз описана геометрія конектора.

У твоєму поточному варіанті сині точки не мають **стабільної прив’язки до конкретного сегмента**. Вони кожен раз заново обчислюються із `connector.geometry.points`, а сам рух зберігається лише як один глобальний `baseOffset`. Через це після першої ж перебудови ти вже редагуєш не “цей самий сегмент”, а нову геометрію, де набір сегментів змінився. Саме тому після зміщення горизонтального сегмента виникає хаос: з’являється ще один вертикальний сегмент, синя точка “переїжджає”, а далі різні сегменти починають боротися між собою. Це прямо видно з твого коду: `getConnectorSegmentHandles` кожен раз бере всі сегменти з `geometry.points`, `moveElbowSegmentHandle` зберігає тільки один `baseOffset`, а `buildOrthogonalElbowPoints` перебудовує всю ламану цілком із цього одного параметра. 

### У чому коренева причина

Зараз у тебе модель така:

* є `start`, `end`
* є `route.baseOffset`
* з цього щоразу будується весь шлях

Це працює тільки для дуже простого стану: “один L-подібний конектор”. Але щойно ти рухаєш горизонтальний сегмент, шлях уже перестає бути просто L-подібним. Він стає ламаною з **трьома сегментами**:

* ліва вертикаль
* середня горизонталь
* права вертикаль

Після цього одного `baseOffset` вже недостатньо.

Тобто проблема не в `dragBoundFunc` і не в самому `Rect`, а в тому, що:

1. ти редагуєш **не сегмент**, а глобальний параметр;
2. після перебудови список сегментів змінюється;
3. сині точки генеруються заново вже по новій геометрії;
4. але логіка руху не знає, **який саме сегмент був вихідним**. 

---

# Що треба змінити концептуально

Тобі треба перейти з моделі:

`start + end + baseOffset`

на модель:

`start + end + список ортогональних вершин (vertices / bend points)`

Тобто конектор має зберігатися не як “формула”, а як **стабільний маршрут**.

Наприклад так:

```js
connector.route = {
  vertices: [
    { x: 420, y: 120, id: "v1" },
    { x: 420, y: 220, id: "v2" }
  ]
}
```

Або навіть без `id`, але краще з `id` для стабільності.

Тоді повний шлях будується так:

* початок `start`
* далі всі `vertices`
* кінець `end`

І при цьому:

* жовті точки = вершини (`vertices`)
* сині точки = handles сегментів, які обчислюються між сусідніми точками маршруту

Але важливо: **рух синьої точки не повинен перебудовувати маршрут з нуля**. Він має змінювати **лише конкретний сегмент** у списку точок.

---

# Чому зараз вертикальний сегмент працює “майже нормально”

Бо коли ти тягнеш вертикальний сегмент, ти по суті міняєш `x` всього зламу, і це ще вкладається в модель `baseOffset.x`. У твоєму `buildOrthogonalElbowPoints` це та гілка:

```js
if (baseOffset?.axis === "x") {
  return normalizePoints([
    start.x, start.y,
    baseOffset.value, start.y,
    baseOffset.value, end.y,
    end.x, end.y,
  ]);
}
```

Тобто геометрія все ще описується одним “коліном”. Тому здається, що все ок. Але коли рухаєш горизонтальний сегмент, логіка вже переходить у стан, де фактично існує три сегменти, а модель усе ще думає, що в неї тільки одне “коліно”. 

---

# Як має працювати правильно

## Сценарій 1. Початковий simple elbow

Було:

```text
start ─────┐
           │
           end
```

Тут є:

* 1 горизонтальний сегмент
* 1 вертикальний сегмент

Сині точки:

* одна по центру горизонтального
* одна по центру вертикального

### Якщо тягнемо вертикальний сегмент вліво/вправо

Має змінитися тільки `x` цього сегмента.

Результат:

```text
start ────┐
          │
          │
          end
```

Просто інше положення вертикалі.

---

## Сценарій 2. Тягнемо горизонтальний сегмент вгору/вниз

Було:

```text
start ─────┐
           │
           end
```

Після перетягування горизонтального сегмента вниз має стати:

```text
start
│
│
└──────────┐
           │
           end
```

Тобто:

* старий горизонтальний сегмент не “зникає”
* він перетворюється на **середній горизонтальний сегмент**
* додається **нова вертикаль зліва**
* стара вертикаль справа лишається

І тепер сині точки мають бути:

* на лівій вертикалі
* на середній горизонталі
* на правій вертикалі

Ключове: кожна з них редагує **тільки свій сегмент**.

---

## Сценарій 3. Далі тягнемо ліву вертикаль

Має рухатися тільки ліва вертикаль.
Середній горизонтальний сегмент просто подовжується або скорочується.
Права вертикаль не повинна “підхоплюватися”.

---

## Сценарій 4. Тягнемо середній горизонтальний ще раз

Має змінюватись тільки його `y`.
Дві вертикалі залишаються вертикалями й просто перераховують довжину.

---

# Яка модель даних потрібна

Ось мінімально нормальна модель:

```js
route: {
  vertices: [
    { id: "v1", x: end.x, y: start.y }
  ]
}
```

Для простого elbow достатньо однієї вершини.

Але для orthogonal polyline краще одразу так:

```js
route: {
  points: [
    { x: start.x, y: start.y, kind: "anchor-start" },
    { x: end.x, y: start.y, kind: "bend", id: "b1" },
    { x: end.x, y: end.y, kind: "anchor-end" }
  ]
}
```

Або окремо:

* `anchors` — start/end, вони обчислюються
* `vertices` — тільки внутрішні точки

Я би радив такий формат:

```js
route: {
  vertices: [
    { id: "b1", x: end.x, y: start.y }
  ]
}
```

А повний список точок щоразу збирати так:

```js
[start, ...vertices, end]
```

---

# Як перебудувати логіку

## 1. `buildOrthogonalElbowPoints` більше не має працювати через `baseOffset`

Замість цього він має будувати шлях із вершин:

```js
export function buildOrthogonalRoutePoints(start, end, route = {}) {
  const vertices = route?.vertices || [];

  const full = [start, ...vertices, end];
  const points = [];

  for (let i = 0; i < full.length; i++) {
    points.push(full[i].x, full[i].y);
  }

  return normalizePoints(points);
}
```

Але тут важливо, щоб `vertices` уже були ортогонально правильними.

---

## 2. Початковий simple elbow треба створювати не через `baseOffset`, а через 1 вершину

Наприклад:

```js
export function createInitialElbowRoute(start, end) {
  return {
    vertices: [
      {
        id: crypto.randomUUID(),
        x: end.x,
        y: start.y,
      },
    ],
  };
}
```

Це одразу дає тобі маршрут:

```text
start -> bend -> end
```

---

## 3. Сині точки повинні будуватись не по “абстрактних сегментах geometry”, а по стабільних парах точок маршруту

Наприклад:

```js
export function getConnectorSegmentHandles(connector) {
  if (!connector || connector.kind !== "connector") return [];
  if (connector.pathType !== "orthogonal-elbow") return [];

  const start = connector.geometry?.start;
  const end = connector.geometry?.end;
  const vertices = connector.route?.vertices || [];

  const pts = [start, ...vertices, end];
  const handles = [];

  for (let i = 0; i < pts.length - 1; i++) {
    const a = pts[i];
    const b = pts[i + 1];

    const isHorizontal = a.y === b.y;
    const isVertical = a.x === b.x;
    if (!isHorizontal && !isVertical) continue;

    handles.push({
      id: `seg-${connector.id}-${i}`,
      segmentIndex: i,
      orientation: isHorizontal ? "horizontal" : "vertical",
      x: (a.x + b.x) / 2,
      y: (a.y + b.y) / 2,
    });
  }

  return handles;
}
```

Тут `segmentIndex` уже стабільно означає:

* сегмент між точками `pts[i]` і `pts[i + 1]`

---

## 4. Перетягування синьої точки має змінювати **локально** тільки одну пару точок

Оце найважливіше.

### Якщо тягнемо внутрішній горизонтальний сегмент

Треба змінити `y` у двох точок, що його обмежують.

### Якщо це крайній сегмент

Можливо, треба вставити нову вершину.

---

# Практична стратегія, яка реально спрацює

Я раджу не стрибати одразу в “універсальний редактор”. Зроби 3 етапи.

---

## Етап A. Стабільний simple elbow без хаосу

На цьому етапі:

* зберігаєш `route.vertices`
* стартово маєш 1 вершину
* сині точки є на 2 сегментах
* рух будь-якого синього сегмента **не створює нові сегменти автоматично**, а лише змінює існуючу вершину

### Як це виглядатиме

Було:

```text
start -> bend -> end
```

де `bend = { x: end.x, y: start.y }`

#### Рух вертикального сегмента

міняє `bend.x`

#### Рух горизонтального сегмента

міняє `bend.y`

Оце вже прибере дригання повністю, бо в тебе буде одна стабільна вершина, а не `baseOffset`, який щоразу генерує новий shape.

---

## Етап B. Дозволити горизонтальному сегменту створювати “сходинку”

Оце вже той сценарій, який ти хочеш.

Коли юзер тягне крайній горизонтальний сегмент, ти не просто міняєш `bend.y`, а перетворюєш simple elbow у polyline з 2 вершинами.

Було:

```js
vertices = [{ id: "b1", x: end.x, y: start.y }]
```

Стало після drag горизонтального сегмента вниз:

```js
vertices = [
  { id: "b1", x: start.x, y: newY },
  { id: "b2", x: end.x, y: newY }
]
```

Геометрія стане:

```text
start
↓
(start.x, newY) ───── (end.x, newY)
                               ↓
                              end
```

Це саме те, що ти описував.

---

## Етап C. Повноцінний polyline editor

Після цього будь-який сегмент редагується локально:

* рухаєш вертикальний сегмент між `p[i]` і `p[i+1]`

  * міняється `x` у цих двох точок
* рухаєш горизонтальний

  * міняється `y` у цих двох точок

Якщо сегмент крайній, логіка трохи окрема.
Якщо внутрішній — усе просто.

# Чому зараз дригається

Бо `onDragMove` кожного разу:

1. бере `connector`
2. змінює його через `moveElbowSegmentHandle`
3. викликає `rebuildConnectorGeometry`
4. `replaceEdge`

Після цього:

* `geometry.points` змінилися
* `getConnectorSegmentHandles()` повернув уже інший набір
* позиція `Rect` та сегментів стала іншою

Тобто під час одного drag у тебе сам об’єкт редагування постійно змінює свою топологію. Це дуже нестабільно. Усе це випливає з поточної схеми `baseOffset + regenerate all`.

# Що я раджу зробити прямо зараз

## Варіант, який найшвидше дасть стабільність

### Крок 1. Повністю прибрати `baseOffset`

і перейти на:

```js
route.vertices
```

### Крок 2. Для simple elbow мати рівно одну вершину

```js
vertices = [{ id, x: end.x, y: start.y }]
```

### Крок 3. `getConnectorSegmentHandles()` будувати від `[start, ...vertices, end]`

### Крок 4. `moveElbowSegmentHandle()` переписати так, щоб він працював через `segmentIndex`

# Приклад логіки `moveElbowSegmentHandle`

Ось хороша базова схема.

```js
export function moveElbowSegmentHandle(connector, handle, pos) {
  if (!connector || connector.kind !== "connector") return connector;
  if (connector.pathType !== "orthogonal-elbow") return connector;

  const start = connector.geometry?.start;
  const end = connector.geometry?.end;
  const vertices = [...(connector.route?.vertices || [])];

  const pts = [start, ...vertices, end];
  const i = handle.segmentIndex;

  if (i < 0 || i >= pts.length - 1) return connector;

  const a = pts[i];
  const b = pts[i + 1];

  const isHorizontal = a.y === b.y;
  const isVertical = a.x === b.x;

  if (!isHorizontal && !isVertical) return connector;

  // special case: simple elbow, 1 vertex
  if (vertices.length === 1) {
    const bend = { ...vertices[0] };

    if (isHorizontal) {
      bend.y = pos.y;
    } else if (isVertical) {
      bend.x = pos.x;
    }

    return {
      ...connector,
      route: {
        ...connector.route,
        vertices: [bend],
      },
    };
  }

  // polyline case
  const internalIndexA = i - 1;     // index in vertices if a is internal
  const internalIndexB = i;         // index in vertices if b is internal

  if (isHorizontal) {
    if (internalIndexA >= 0) vertices[internalIndexA] = { ...vertices[internalIndexA], y: pos.y };
    if (internalIndexB < vertices.length) vertices[internalIndexB] = { ...vertices[internalIndexB], y: pos.y };
  }

  if (isVertical) {
    if (internalIndexA >= 0) vertices[internalIndexA] = { ...vertices[internalIndexA], x: pos.x };
    if (internalIndexB < vertices.length) vertices[internalIndexB] = { ...vertices[internalIndexB], x: pos.x };
  }

  return {
    ...connector,
    route: {
      ...connector.route,
      vertices,
    },
  };
}
```

Це вже дає тобі стабільну поведінку без “перескакування”.

---

# Як зробити утворення нового сегмента

Це окрема дія. Її не треба змішувати з поточною логікою `baseOffset`.

Для simple elbow з 1 вершиною:

```js
vertices = [{ x: end.x, y: start.y }]
```

Якщо юзер тягне горизонтальний сегмент і ти хочеш утворити нову “сходинку”, тоді замість зміни однієї вершини ти робиш split:

```js
vertices = [
  { id: "b1", x: start.x, y: newY },
  { id: "b2", x: end.x, y: newY }
];
```

Якщо тягне вертикальний:

```js
vertices = [
  { id: "b1", x: newX, y: start.y },
];
```

Тобто створення нового сегмента — це **контрольована операція split**, а не побічний ефект `rebuildConnectorGeometry`.

---

# Що ти отримаєш у результаті

Після переходу на `vertices`:

### 1. Зникне дригання

Бо handle буде прив’язаний до `segmentIndex`, а не до “тимчасової геометрії”.

### 2. Кожен сегмент стане локально керованим

Тягнеш конкретний сегмент — міняється тільки він.

### 3. Зможеш нормально додавати нові злами

Через явний split маршруту.

### 4. Логіка стане масштабованою

Тобто потім можна буде:

* видаляти зайві злами
* merge колінеарних сегментів
* підтримувати waypoint-и
* снап до сітки
* перевірку перетинів

---

# Як я б радив впроваджувати це по факту

Не намагайся зараз “підлатати” `baseOffset`. Його краще прибрати, бо він концептуально не тягне polyline-редактор.

Послідовність така:

## Фаза 1

Переписати `orthogonal-elbow` на `route.vertices` з однією вершиною.

## Фаза 2

Переписати `getConnectorSegmentHandles()` через маршрут `[start, ...vertices, end]`.

## Фаза 3

Переписати `moveElbowSegmentHandle()` так, щоб він міняв вершини локально.

## Фаза 4

Додати `splitSegmentForDrag()` — коли треба перетворити simple elbow у складнішу ламану.

## Фаза 5

Додати `normalizeOrthogonalVertices()`:

* прибирати дублікати
* зливати колінеарні точки
* не допускати нульових сегментів

# Висновок

Головна проблема зараз така:

**у тебе немає стабільної моделі маршруту, є тільки формульна перебудова всієї лінії через один `baseOffset`**. Через це код не може коректно підтримувати сценарій, де після drag одного сегмента з’являються нові сегменти і кожен із них далі редагується окремо. Це прямо видно з того, що `getConnectorSegmentHandles()` бере всі сегменти з поточної геометрії, а `moveElbowSegmentHandle()` не знає нічого про конкретний сегмент і зберігає лише глобальний offset. 

Тому правильне виправлення — не “ще підкрутити if”, а перейти на модель `route.vertices`.

Я можу в наступному повідомленні вже дати тобі готовий робочий набір функцій для:

* `createInitialElbowRoute`
* `buildOrthogonalRoutePoints`
* `getConnectorSegmentHandles`
* `moveElbowSegmentHandle`
* `normalizeOrthogonalVertices`

у форматі під твій current React/Konva store.
