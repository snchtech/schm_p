Так, ти правильно бачиш ризик: якщо в майбутньому зміниться логіка `buildEdgePath`, а стара схема відкриється в режимі редагування, то перший же drag вузла може перебудувати всі `points` по-новому і візуально “зламати” стару схему.

І це якраз типова задача **evolution of document model**: треба проєктувати не тільки редактор, а й **сумісність формату схем** між поколіннями.

Найкраща практика тут не в тому, щоб тримати 20 копій функцій промальовки, а в тому, щоб розділити:

1. **формат документа схеми**
2. **layout/routing engine**
3. **режим сумісності**
4. **міграції**
5. **політику редагування старих схем**

Із цього виходить дуже здорова архітектура.

---

# 1. Головний принцип

Тобі не треба версіонувати кожну функцію окремо.
Треба версіонувати **схему як документ** і **движок її інтерпретації**.

Тобто не так:

* `buildEdgePathV1`
* `buildEdgePathV2`
* `buildEdgePathV3`
* `drawNodeV4`

а так:

* у схеми є `documentVersion`
* у зв’язку є `routingMode`
* редактор знає, як поводитись із документами певної версії
* старий документ можна:

  * показувати як є
  * частково редагувати
  * або відкривати через “upgrade to new version”

Це значно чистіше.

---

# 2. Найкраще рішення: розділити “дані” і “побудову”

Одна з найважливіших best practice:

> **не зберігати тільки похідну геометрію, якщо вона може бути перебудована новим алгоритмом**

Але так само:

> **не покладатися тільки на новий алгоритм для старих документів**

Тобто для edge треба розрізняти 2 рівні:

---

## Рівень A. Семантична модель edge

Що з чим з’єднано

```js
{
  id: "e1",
  source: { type: "node", nodeId: "n1", direction: "right" },
  target: { type: "node", nodeId: "n2", direction: "left" },
  routing: {
    mode: "orthogonal",
    startOffset: 20,
    endOffset: 20
  }
}
```

---

## Рівень B. Геометричний snapshot

Як саме це було намальовано в цій версії

```js
{
  points: [100, 50, 140, 50, 140, 90, 220, 90],
  handles: [...],
  anchors: [...]
}
```

---

Оце і є ключ.
Для кожного edge можна зберігати:

* **semantic source/target**
* **routing params**
* **render snapshot**

Тоді стару схему можна відкривати **без перебудови**.

---

# 3. Найздоровіша модель: dual representation

Я би радив таку архітектуру edge:

```js
{
  id: "e1",

  source: { ... },
  target: { ... },

  routing: {
    engine: "orthogonal-v1",
    params: {
      startOffset: 20,
      endOffset: 20
    }
  },

  geometry: {
    points: [...],
    anchors: [...],
    elbows: [...]
  }
}
```

Сенс такий:

* `source/target` — логіка
* `routing.engine` — який engine породив маршрут
* `routing.params` — параметри engine
* `geometry` — зафіксований результат

---

## Чому це добре

Бо тоді ти можеш:

### для старих схем

відображати `geometry.points` як є, взагалі не чіпаючи новий routing

### для нових схем

будувати `geometry` через новий engine

### при редагуванні

сам вирішувати:

* чи edge треба залишити frozen
* чи перебудувати
* чи запропонувати upgrade

---

# 4. Найкраща стратегія сумісності

Тут є 3 нормальні стратегії.

---

## Стратегія 1. Full backward compatibility engine

Тримати старі engine

Наприклад:

* `orthogonal-v1`
* `orthogonal-v2`
* `smart-router-v3`

І при rebuild брати саме той engine, який записаний у документі.

### плюси

* старі схеми поводяться як раніше
* можна редагувати без поломки

### мінуси

* треба підтримувати старі routing engine

### коли підходить

Коли різниця між версіями routing справді суттєва.

---

## Стратегія 2. Frozen legacy documents

Старі схеми не перебудовуються автоматично

Тобто:

* стара схема відкривається
* `points` беруться із збереженого snapshot
* можна дивитися, можливо рухати whole block
* але structural editing вимикається
* для редагування треба створити нову версію документа

### плюси

* дуже безпечно
* старі схеми не ламаються

### мінуси

* редагування старих схем обмежене

### коли підходить

Коли ти різко міняєш модель зв’язків.

---

## Стратегія 3. Upgrade on edit

Поки документ просто відкритий — показуєш як є.
Коли користувач хоче редагувати структуру, робиш upgrade.

Наприклад:

* схема v1
* користувач тягне node
* система каже:
  “Ця схема створена старим движком. Для редагування потрібно конвертувати у новий формат. Створити нову версію?”

### плюси

* чистий UX
* стара схема залишається недоторканою
* нова версія вже працює на новому engine

### мінуси

* треба робити migration flow

### коли підходить

Це, на мою думку, **найкращий варіант для тебе**.

---

# 5. Що я рекомендую саме для твого кейсу

Для такого редактора я би радив гібрид:

## базове правило

**Стара схема повинна відкриватися без автоматичного recompute edge geometry.**

Тобто:

* не відкривати стару схему і одразу проганяти через новий `buildEdgePath`
* не перераховувати `points`, якщо користувач просто переглядає

## далі

Кожна схема має:

* `schemaVersion`
* `routingEngineVersion`

Наприклад:

```js
{
  id: "scheme-17",
  schemaVersion: 1,
  routingEngineVersion: "orthogonal-v1",
  ...
}
```

## при відкритті

редактор дивиться:

* якщо `routingEngineVersion` підтримується — можна редагувати в legacy mode
* якщо ні або ти не хочеш підтримувати — відкриває read-only / limited-edit mode

## при спробі structural edit

створюється нова версія схеми:

* `scheme-17` лишається як є
* `scheme-17-v2` або новий revision
* edges конвертуються / перебудовуються під новий engine

Оце найздоровіше.

---

# 6. Не версіонувати функції — версіонувати capability

Тобі не треба 20 функцій `drawEdgeV*`.
Треба зробити registry engine-ів.

Наприклад:

```js
const routingRegistry = {
  "orthogonal-v1": orthogonalRouterV1,
  "orthogonal-v2": orthogonalRouterV2,
  "smart-v1": smartRouterV1,
};
```

І далі edge rebuild робиться не напряму через один `buildEdgePath`, а так:

```js
function rebuildEdge(edge, context) {
  const engineId = edge.routing?.engine || context.defaultEngine;
  const router = routingRegistry[engineId];

  if (!router) {
    return edge.geometry?.points || [];
  }

  return router.build(edge, context);
}
```

Тобто ти підтримуєш не 20 розкиданих функцій, а **registry маршрутизаторів**.

Це вже дуже дорослий підхід.

---

# 7. Схема повинна мати revision model

Я би радив не просто “зберігати схему”, а мати:

* `scheme_id` — логічна схема
* `revision_id` — конкретна редакція

Наприклад:

```js
SCHEME
- id
- title
- current_revision_id

SCHEME_REVISION
- id
- scheme_id
- version_no
- schema_version
- routing_engine_version
- created_at
- created_by
- parent_revision_id
- is_editable
- snapshot_json
```

Тоді можна:

* лишити стару редакцію недоторканою
* створити нову редакцію після migration
* дозволити rollback

Це значно краще, ніж просто “перезаписати JSON”.

---

# 8. Як краще поводитися зі старою схемою

Ось найпрактичніша політика.

---

## Для старих ревізій:

### Дозволити:

* перегляд
* виділення
* масштабування
* панорамування
* відкриття властивостей
* експорт / друк
* можливо редагування тексту/метаданих

### Обмежити або вимкнути:

* drag node, якщо це призведе до recompute edge
* reconnect edge
* add edge
* delete anchor
* auto-layout

### Показати:

“Ця схема створена старим форматом. Для редагування структури створіть нову редакцію.”

Це дуже здоровий UX.

---

# 9. Коли варто робити міграцію, а коли ні

Не всі зміни потребують нової версії engine.

---

## Не потребують engine version bump:

* зміна кольору лінії
* нові hover-ефекти
* контекстне меню
* selection logic
* UI handle size
* label rendering, якщо geometry не змінюється

---

## Потребують engine version bump:

* зміна алгоритму orthogonal routing
* зміна правил обходу вузлів
* зміна логіки `resolveDirection`
* нова модель anchor-ів
* edge-to-edge semantics
* інша система offsets/elbows
* автоматичне перепроєктування середніх точок

Тобто version bump має стосуватися не “будь-якої зміни коду”, а **зміни контракту геометрії**.

---

# 10. Що саме зберігати в JSON схеми

Я б радив зберігати так:

```js
{
  schemaVersion: 2,
  routingEngineVersion: "orthogonal-v2",

  nodes: [...],

  edges: [
    {
      id: "e1",

      source: { ... },
      target: { ... },

      routing: {
        engine: "orthogonal-v2",
        params: {
          startOffset: 20,
          endOffset: 20
        }
      },

      geometry: {
        points: [...],
        anchors: [...],
        handles: [...]
      },

      meta: {
        createdWithEditor: "2.3.0",
        upgradedFrom: "orthogonal-v1"
      }
    }
  ]
}
```

---

# 11. Як відкривати документ у редакторі

Правильний flow відкриття:

### крок 1

читаєш `schemaVersion`, `routingEngineVersion`

### крок 2

вирішуєш policy:

* editable
* limited-edit
* read-only
* needs-upgrade

### крок 3

у canvas грузиш:

* `nodes`
* `edges.geometry.points`
* `anchors`
* metadata

### крок 4

не перебудовуєш edges автоматично, поки цього не вимагає конкретна action

Оце дуже важливо.

---

# 12. Який режим rebuild має бути

Тобі не можна мати один універсальний сценарій:

```js
onNodeDrag -> always rebuild all connected edges
```

Це небезпечно для legacy docs.

Потрібна політика:

```js
if (document.editMode === "legacy-frozen") {
  blockNodeDragRebuild();
}

if (document.editMode === "legacy-compatible") {
  rebuildWithLegacyEngine();
}

if (document.editMode === "current") {
  rebuildWithCurrentEngine();
}
```

Тобто rebuild залежить не тільки від edge, а й від **режиму документа**.

---

# 13. Найкраща модель режимів документа

Я б зробив так:

### `current`

Повне редагування, поточний engine

### `legacy-compatible`

Редагування дозволене, але тільки через старий engine

### `legacy-frozen`

Старий snapshot, structural edit заблокований

### `upgrade-pending`

Документ відкритий, але для редагування треба migration

Це набагато краще, ніж просто “стара/нова схема”.

---

# 14. Міграції повинні бути явними

Дуже важлива best practice:

> міграції формату документа мають бути явними, а не прихованими

Тобто не так:

* відкрив стару схему
* трохи посунув блок
* вона вже безшумно збереглась у новому форматі

А так:

* відкрив стару схему
* система бачить старий engine
* пропонує створити нову ревізію
* запускає migration
* ти працюєш уже з новою копією

Так безпечніше і зрозуміліше.

---

# 15. Як технічно зробити migration

При міграції стара схема лишається незмінною.
Нова ревізія створюється так:

### Вхід:

* nodes
* edges old format
* old geometry snapshot

### Процес:

* перетворення edge model у новий shape
* нормалізація anchor-ів
* визначення `routing.engine = "orthogonal-v2"`
* опціонально:

  * або взяти старі points як стартовий snapshot
  * або перебудувати весь маршрут новим engine

### Вихід:

* новий revision JSON

---

# 16. Чи треба забороняти редагування старої версії?

У більшості випадків — **так, структурне редагування краще не дозволяти**.

Тобто:

* перегляд — так
* змінити назву/опис/метадані — так
* посунути node, що перебудує зв’язки — ні
* додати новий edge — ні
* видалити edge-anchor — ні

Бо як тільки ти дозволив structural editing старої схеми, ти вже або:

* мусиш підтримувати старий engine чесно
* або ризикуєш поламати layout

Якщо не хочеш нести тягар повної backward-compatible routing-логіки, тоді краще блокувати таке редагування.

---

# 17. Найкраща практична рекомендація для тебе

Ось що я б радив як реальний план.

---

## Рішення

### 1.

Ввести у схему:

* `schemaVersion`
* `routingEngineVersion`
* `editPolicy`

### 2.

Для кожного edge зберігати:

* semantic model
* routing params
* geometry snapshot

### 3.

При відкритті:

* не перебудовувати автоматично старі `points`

### 4.

Для старих схем:

* або `legacy-frozen`
* або `legacy-compatible`

### 5.

При серйозних змінах routing:

* не переписувати старі ревізії
* створювати нову ревізію схеми

### 6.

Побудувати registry routing engine-ів, а не хаотичний набір `V1/V2/V3` функцій по всьому коду

---

# 18. Що я вважаю оптимальним саме для тебе

З урахуванням твого кейсу, я б обрав таку політику:

## Для вже створених схем:

* зберігати як snapshot + metadata version
* відкривати без recompute
* structural edit не дозволяти напряму

## Для нових схем:

* використовувати новий routing engine
* повне редагування

## Якщо треба змінити стару схему:

* кнопка **“Створити нову редакцію”**
* migration у новий формат
* редагування вже нової ревізії

Це дуже практично і безпечно.

---

# 19. Якщо коротко: як краще зробити

Найкращий підхід такий:

> **не підтримувати 20 розкиданих версій функцій, а підтримувати версію документа і невеликий registry routing engine-ів; старі схеми відкривати без автоматичного recompute, а для змін створювати нову ревізію**

Це дає тобі:

* стабільність старих схем
* можливість еволюції редактора
* контрольовані міграції
* чистішу архітектуру
* нормальну підтримку в довгу

---

# 20. Моя пряма рекомендація

Якщо зовсім конкретно:
**не редагуй старі схеми тим самим live-routing-пайплайном, що й нові.**
Роби одну з двох речей:

* або legacy mode з конкретним engine
* або read-only / limited edit + “create new revision”

Для твого проєкту другий варіант виглядає найздоровішим.

Наступним кроком я можу тобі запропонувати **конкретну структуру JSON/таблиць БД для versioned scheme + scheme_revision + edge snapshot**, щоб це одразу лягло на твою архітектуру.
