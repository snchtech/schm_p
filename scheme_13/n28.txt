Так, це добре вкладається в твою поточну архітектуру. І тут головна порада така:

> **не робити окремо “лінію”, “стрілку”, “подвійну стрілку”, “стрілку-перехід” як зовсім різні сутності**
> а зробити **один connector-об’єкт** і дати йому:
>
> * `pathType`
> * `endcaps`
> * `behavior/meta`

Тоді коду не буде розростатись лавиною.

---

# 1. Як я б це мислив архітектурно

У тебе вже є хороша база:

* `nodes`
* `edges`
* новий `kind: "connector"`

Оце і треба лишити.

Тепер connector має описуватись не тільки як:

* `orthogonal-straight`
* `orthogonal-elbow`

а більш універсально.

Я б радив розділити connector на 3 незалежні властивості:

## A. Тип траєкторії

Як саме будується лінія

Наприклад:

* `orthogonal-straight`
* `orthogonal-elbow`
* `free-straight` або `angled-straight`

## B. Оформлення кінців

Що є на початку і в кінці

Наприклад:

* `none`
* `arrow`
* `diamond`
* `circle`

## C. Поведінка / призначення

Що це за тип зв’язку з точки зору бізнес-логіки

Наприклад:

* `default`
* `navigation`
* `signal`
* `reference`

Оце дуже важливо, бо “стрілка для переходу” — це не стільки інша геометрія, скільки **інша поведінка**.

---

# 2. Яку модель я б радив

Не так:

* `arrow-line`
* `double-arrow-line`
* `navigation-arrow`
* `transition-arrow`

А так:

```js id="s437s6"
{
  id: "...",
  kind: "connector",

  pathType: "free-straight", // orthogonal-straight | orthogonal-elbow | free-straight

  start: { ... },
  end: { ... },

  route: {
    waypoints: [],
    baseOffset: null,
  },

  endcaps: {
    start: "none",   // none | arrow
    end: "arrow",    // none | arrow
  },

  behavior: {
    type: "navigation", // default | navigation
    link: {
      targetSchemeId: "...",
      highlightParam: "...",
    },
  },

  geometry: {
    points: [],
  },

  style: {
    stroke: "black",
    strokeWidth: 2,
  },
}
```

---

# 3. Чому це найкраще рішення

Бо тоді:

## проста лінія

```js id="6od1n3"
pathType: "free-straight",
endcaps: { start: "none", end: "none" }
```

## стрілка в один бік

```js id="q0wpf1"
pathType: "free-straight",
endcaps: { start: "none", end: "arrow" }
```

## стрілка в два боки

```js id="zvjt04"
pathType: "free-straight",
endcaps: { start: "arrow", end: "arrow" }
```

## ортогональна лінія зі стрілкою

```js id="n8l2e4"
pathType: "orthogonal-elbow",
endcaps: { start: "none", end: "arrow" }
```

## стрілка-перехід

```js id="6i1iy4"
pathType: "free-straight",
endcaps: { start: "none", end: "arrow" },
behavior: {
  type: "navigation",
  link: {
    targetSchemeId: "scheme-15",
    highlightParam: "pump-17"
  }
}
```

Тобто геометрія, оформлення і бізнес-сенс не змішуються в одну купу.

---

# 4. Для твого нового типу “лінія зі стрілкою під будь-яким кутом”

Це я б назвав:

```js id="e0rew7"
pathType: "free-straight"
```

або

```js id="0v4dqh"
pathType: "angled-straight"
```

Суть:

* це просто один сегмент
* не ортогональний
* може бути під будь-яким кутом
* start/end тягнуться як завгодно

Тобто тут уже не треба ортогональна математика.
`geometry.points` буде просто:

```js id="ko6409"
[start.x, start.y, end.x, end.y]
```

# 5. Як це вписати в інструменти

Я б не робив кнопки на рівні:

* “стрілка вправо”
* “стрілка вліво”

Це зайве. Бо напрямок стрілки природно виходить із `start/end`.

## Краще так:

### Кнопка 1

**Пряма лінія**

* `pathType: "free-straight"`
* `endcaps.start = "none"`
* `endcaps.end = "none"`

### Кнопка 2

**Стрілка**

* `pathType: "free-straight"`
* `endcaps.start = "none"`
* `endcaps.end = "arrow"`

### Кнопка 3

**Подвійна стрілка**

* `pathType: "free-straight"`
* `endcaps.start = "arrow"`
* `endcaps.end = "arrow"`

### Кнопка 4

**Ламана**

* `pathType: "orthogonal-elbow"`
* `endcaps.start = "none"`
* `endcaps.end = "none"`

### Кнопка 5

**Ламана зі стрілкою**

* `pathType: "orthogonal-elbow"`
* `endcaps.start = "none"`
* `endcaps.end = "arrow"`

Тобто ти комбінуєш:

* траєкторію
* наконечники

а не множиш типи без потреби.

---

# 6. Як зробити, щоб стрілка “змінила напрямок”, якщо її перетягнути

Це вже природно випливає з геометрії.

Якщо:

* `start` зліва, `end` справа — стрілка дивиться вправо
* якщо користувач перетягнув `end` вліво від `start` — стрілка автоматично дивиться вліво

Тобто не треба окрема логіка “стрілка вправо/вліво”.
Потрібно лише в render обчислювати кут наконечника за останнім сегментом.

---

# 7. Як технічно рендерити стрілку

У `react-konva` є 2 хороші варіанти:

## Варіант A

Використовувати `Arrow` для простих straight-arrow

## Варіант B

Рендерити `Line` + окремий наконечник стрілки вручну

Я б для твоєї архітектури радив **варіант B**.

Чому:

* один rendering pipeline для всіх connector-ів
* однаковий контроль над:

  * straight
  * elbow
  * single arrow
  * double arrow
* легше підтримувати `endcaps.start/end`

Тобто:

* сам path малюєш `Line`
* наконечники стрілок — окремими shape/полігонами на start/end

---

# 8. Найважливіша архітектурна порада

Не зашивай “стрілку” в `pathType`.

Неправильно:

```js id="ifdy7u"
pathType: "arrow-line"
```

Правильно:

```js id="2xi3j1"
pathType: "free-straight",
endcaps: {
  start: "none",
  end: "arrow"
}
```

Бо інакше ти дуже швидко отримаєш:

* arrow-line
* double-arrow-line
* navigation-arrow-line
* elbow-arrow-line
* elbow-double-arrow-line

І це почне розростатися хаотично.

---

# 9. Що робити зі “стрілкою переходу”

Тут теж не треба робити окремий geometry type.

Стрілка переходу — це просто connector зі звичайною геометрією, але з додатковими метаданими:

```js id="cw04lu"
behavior: {
  type: "navigation",
  link: {
    targetSchemeId: "scheme-15",
    highlightParam: "pump-17",
  }
}
```

Тобто:

## візуально

вона може бути такою ж, як звичайна стрілка

## функціонально

по кліку або в режимі viewer:

* виконує перехід
* передає параметри
* інша схема підсвічує потрібний об’єкт

# 14. Дуже важливий практичний висновок

Тобі не треба робити окремий “елемент стрілка”.
Треба зробити **connector з варіативними наконечниками і поведінкою**.

Тобто стрілка — це не нова базова сутність, а **варіант connector-а**.

Це найкраще з точки зору:

* підтримки
* масштабування
* UX
* уникнення дублювання коду
