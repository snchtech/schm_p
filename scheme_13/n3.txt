Якщо **прямо “залишити як зараз”**, тобто:

* отримати CSV зі списком ID,
* передати його в `apex_util.ir_filter` / `APEX_IR.ADD_FILTER`,
* і далі фільтрувати IR через службовий механізм APEX,

то **надійного рішення тут практично немає**. `IR_FILTER` у сучасному APEX позначений як deprecated, а Oracle рекомендує `APEX_IR`; обидва механізми працюють через внутрішні налаштування Interactive Report, а не як “просто підставити ваш рядок у WHERE”. Саме тому ви і впираєтесь у внутрішній `condition_sql` з лімітом 4000 символів. ([docs.oracle.com][1])

Що можна зробити з **мінімальними змінами**:

### 1. Залишити формування списку ID як є

Тобто ваш `LISTAGG`, page 0, передача значення далі — усе це можна лишити.

### 2. Замінити тільки останній крок

Замість:

```plsql
apex_util.ir_filter(
  p_condition => 'order_id in (' || list || ')'
);
```

зробити фільтрацію в SQL самого Interactive Report.

Наприклад:

```sql
select *
from orders o
where :P0_ORDER_IDS is null
   or o.order_id in (
        select to_number(trim(column_value))
        from apex_string.split(:P0_ORDER_IDS, ',')
   )
```

Це вже не використовує внутрішній великий `condition_sql`, але логіка для вас майже та сама: ви так само формуєте список, так само передаєте його між сторінками, просто **не через API фільтра IR**.

---

## Чи можна лишити саме “візуальний IR filter зверху звіту”

Ось тут уже складніше.

APEX Interactive Report підтримує column і row filters, і Oracle описує їх як стандартний механізм налаштувань звіту. Але це саме **налаштування звіту**, не контейнер для великих довільних `IN (...)` виразів. Тому для довгих списків це місце слабке. ([docs.oracle.com][2])

Тобто:

* **для коротких умов** — так;
* **для довгого списку ID** — технічно може інколи працювати, але стабільно покладатися на це не варто.

---

## Чи є “костиль”, щоб усе ж лишити IR filter

Є лише напівваріанти, але я їх не раджу:

**Розбивати список на шматки**
Наприклад, кілька умов по 30–50 ID. Але IR зазвичай комбінує фільтри як налаштування звіту, і така схема стає крихкою, заплутаною та все одно може впертися в службові межі.

**Пробувати `APEX_IR.ADD_FILTER` замість `APEX_UTIL.IR_FILTER`**
Це правильніше з точки зору API, бо `IR_FILTER` deprecated. Але це не прибирає головної проблеми: фільтр усе одно зберігається як metadata Interactive Report. ([docs.oracle.com][1])

---

## Найближчий до “залишити все як зараз” варіант

Ось що я б назвав **мінімальною переробкою**:

* лишаєте `LISTAGG`;
* лишаєте item на page 0;
* лишаєте передачу цього item в інший додаток;
* **забираєте тільки `ir_filter`**;
* у SQL звіту додаєте умову через `apex_string.split`.

Тобто фактично міняється лише одна точка — спосіб застосування фільтра.

---

## Висновок

**Повністю лишити як є — безпечно не вийде.**
**Майже лишити як є — так, вийде.**

Найменша зміна:

* список ID продовжуєте формувати так само,
* але замість `apex_util.ir_filter(...)` фільтруєте сам SQL регіону.

Це найменш болючий варіант і він не залежить від внутрішнього ліміту `condition_sql` в APEX. ([docs.oracle.com][1])

Можу одразу дати готовий шаблон для вашого кейсу:
**process на page 0 + item + SQL Interactive Report**, щоб ви вставили майже без переробок.
