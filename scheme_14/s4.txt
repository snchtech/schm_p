Добре. Йдемо в **підкрок 2.1**:
**переписуємо `buildOrthogonalElbowPoints()` для одного жовтого path point** так, щоб жовта точка була **центром нового сегмента**, а не вершиною polyline.

Це зараз найважливіше.

---

# 1. Що саме треба змінити в мисленні

Зараз у тебе, ймовірно, логіка така:

* є `start`
* є `end`
* є `waypoint`
* builder робить:

  * `start -> waypoint -> end`

І це неправильно.

## Треба так:

якщо жовта точка додана на сегмент, то вона означає:

* створити **новий внутрішній сегмент**
* поставити жовту **в центр цього сегмента**
* додати **2 точки зламу навколо нього**

---

# 2. Як це виглядає на прикладі горизонтального сегмента

Було:

```text
A ----------------------- B
          (синя)
```

Додали жовту точку.

Треба отримати не:

```text
A ---------- Y ---------- B
```

а щось типу:

```text
A ---------┐
           ├── short horizontal segment ──┐
                                         B
```

І жовта точка стоїть **в центрі цього short horizontal segment**.

---

# 3. Мінімальна модель для одного жовтого path point

Для першого робочого кроку достатньо в `route.waypoints[0]` мати:

```js id="step21-01"
{
  id,
  segmentIndex,
  axis: "y", // якщо додано на горизонтальний сегмент
  x,
  y,
}
```

або

```js id="step21-02"
{
  id,
  segmentIndex,
  axis: "x", // якщо додано на вертикальний сегмент
  x,
  y,
}
```

---

# 4. Головний принцип builder-а

## Якщо `axis === "y"`

це означає:

* path point був доданий на горизонтальний сегмент
* треба створити **новий короткий горизонтальний сегмент** на висоті `waypoint.y`

## Якщо `axis === "x"`

це означає:

* path point був доданий на вертикальний сегмент
* треба створити **новий короткий вертикальний сегмент** на `waypoint.x`

---

# 5. Найпростіший перший алгоритм

Для MVP я раджу не робити ще складний універсальний сегментний engine, а зробити так:

## якщо є тільки 1 жовта точка

і `pathType === "orthogonal-elbow"`

то:

### A. будуємо базову лінію без waypoint

тобто:

* або через `baseOffset`
* або дефолтний elbow

### B. знаходимо сегмент, на який додано path point

### C. замість цього сегмента вставляємо міні-конструкцію з 3 сегментів:

* підхід до нового внутрішнього сегмента
* сам внутрішній сегмент
* вихід із нього

---

# 6. Як саме це виглядає математично

---

## Випадок 1: waypoint на горизонтальному сегменті

Є сегмент:

```js id="step21-03"
(x1, y1) ---- (x2, y2)
```

де `y1 === y2`.

Жовта точка має:

```js id="step21-04"
{ x: wp.x, y: wp.y, axis: "y" }
```

Щоб вона була **центром нового горизонтального сегмента**, беремо якусь довжину нового сегмента, наприклад:

```js id="step21-05"
const INSERTED_SEGMENT_HALF = 20;
```

Тоді новий короткий сегмент буде:

* від:

```js id="step21-06"
(wp.x - 20, wp.y)
```

* до:

```js id="step21-07"
(wp.x + 20, wp.y)
```

І повна вставка замість старого горизонтального сегмента:

```text
(x1, y1)
   -> (wp.x - 20, y1)
   -> (wp.x - 20, wp.y)
   -> (wp.x + 20, wp.y)
   -> (wp.x + 20, y1)
   -> (x2, y1)
```

Оце і є правильна draw.io-подібна логіка.

---

## Випадок 2: waypoint на вертикальному сегменті

Аналогічно.

Було:

```js id="step21-08"
(x1, y1) |
         |
         |
       (x2, y2)
```

де `x1 === x2`.

Жовта точка:

```js id="step21-09"
{ x: wp.x, y: wp.y, axis: "x" }
```

Новий короткий вертикальний сегмент:

* від:

```js id="step21-10"
(wp.x, wp.y - 20)
```

* до:

```js id="step21-11"
(wp.x, wp.y + 20)
```

Повна вставка:

```text
(x1, y1)
  -> (x1, wp.y - 20)
  -> (wp.x, wp.y - 20)
  -> (wp.x, wp.y + 20)
  -> (x1, wp.y + 20)
  -> (x1, y2)
```

---

# 7. Що це вирішує одразу

Ця логіка одразу виправляє твої проблеми:

## Проблема 3

> жовта стає крайовою точкою зламу

Більше не буде.
Бо жовта тепер центр нового внутрішнього сегмента.

## Проблема 4

> жовта накладається на синю

Більше не буде.
Бо старий сегмент замінюється новою структурою, а сині точки потім порахуються заново вже на нових сегментах.

## Проблема 5

> рухаються тільки точки, а не лінія

Більше не буде, якщо builder реально вставляє новий сегмент у geometry.

## Проблема 7

> коли тягну за синю, жовта стає в правий бік

Це теж перестане бути випадковою поведінкою, коли жовта перестане бути vertex polyline.

---

# 8. Як переписати `buildOrthogonalElbowPoints()` на цьому кроці

На кроці 2.1 я б зробив так:

## Логіка функції:

1. якщо `waypoints.length === 0` → стара логіка через `baseOffset`
2. якщо `waypoints.length === 1` → спеціальна логіка вставки внутрішнього сегмента
3. якщо `waypoints.length > 1` → поки можна не підтримувати або лишити fallback

Це нормально для покрокового розвитку.

---

# 9. Який helper варто додати

Я б виніс окремий helper:

```js id="step21-12"
insertPathHandleIntoSegment(points, waypoint)
```

Він:

* бере вже готову базову orthogonal polyline
* знаходить `segmentIndex`
* заміняє один сегмент на конструкцію з 5 точок

Це найчистіший підхід.

---

# 10. Псевдокод цього helper-а

```js id="step21-13"
function insertPathHandleIntoSegment(points, waypoint) {
  const i = waypoint.segmentIndex * 2;

  const x1 = points[i];
  const y1 = points[i + 1];
  const x2 = points[i + 2];
  const y2 = points[i + 3];

  const before = points.slice(0, i + 2);
  const after = points.slice(i + 2);

  if (waypoint.axis === "y") {
    const half = 20;

    const inserted = [
      waypoint.x - half, y1,
      waypoint.x - half, waypoint.y,
      waypoint.x + half, waypoint.y,
      waypoint.x + half, y1,
    ];

    return normalizePoints([
      ...before,
      ...inserted,
      ...after,
    ]);
  }

  if (waypoint.axis === "x") {
    const half = 20;

    const inserted = [
      x1, waypoint.y - half,
      waypoint.x, waypoint.y - half,
      waypoint.x, waypoint.y + half,
      x1, waypoint.y + half,
    ];

    return normalizePoints([
      ...before,
      ...inserted,
      ...after,
    ]);
  }

  return points;
}
```

---

# 11. Як має поводитись drag жовтої точки після цього

Після того як ти змінив `waypoint.y` або `waypoint.x`, builder просто знову викликає `insertPathHandleIntoSegment()`.

Тобто:

* жовта не їздить “по старій лінії”
* а перебудовує свій внутрішній сегмент
* після rebuild жовта знову в центрі цього сегмента

Оце і є draw.io-подібна поведінка.

---

# 12. Що важливо про `segmentIndex`

На першому кроці цього достатньо.
Але знай: після перебудови polyline `segmentIndex` може стати нестабільним.

Для MVP це прийнятно, якщо:

* у тебе поки одна жовта точка
* або ти додаєш їх по одній

Пізніше краще перейти на stable `segmentId`, але зараз це не обов’язково.
