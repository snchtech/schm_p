Так, тут проблема не в одному багу, а в тому, що зараз у тебе **змішані 2 різні сутності в одну модель**:

* **синя точка на відрізку** — це `segment handle`
* **жовта точка шляху** — це `waypoint handle`

А в draw.io вони мають **різну роль**.

---

# Що зараз зламано концептуально

## 1. Синя точка

Повинна:

* завжди бути **в центрі поточного відрізка**
* **не рухатись вздовж лінії**
* рухати **весь відрізок по осі**

  * горизонтальний → тільки `y`
  * вертикальний → тільки `x`
* **не ставати точкою зламу**
* після перебудови знову ставати в центр нового відрізка

Зараз у тебе вона частково поводиться як точка шляху, а не як handle відрізка.

---

## 2. Жовта точка шляху

Повинна:

* створюватись на відрізку окремо
* **не бути крайовою точкою сегмента**
* бути **керуючою точкою нового відрізка**
* після перетягування створювати нову “сходинку”
* теж **не їздити вздовж лінії**
* рухатись тільки по своїй осі:

  * якщо додана на горизонтальному сегменті → рух тільки по `y`
  * якщо на вертикальному → тільки по `x`

Зараз у тебе вона поводиться як вершина polyline, а не як центр керованого сегмента.

---

# Головний висновок

Щоб це працювало як у draw.io, треба перейти до такої моделі:

## Є 2 різні типи керуючих точок

### A. `segmentHandle`

Це синя точка.
Вона не зберігається як “точка зламу”.
Вона лише описує:

* який сегмент редагується
* по якій осі його можна рухати

### B. `pathHandle`

Це жовта точка.
Вона зберігається в `route.waypoints`, але **не як вершина polyline**, а як:

* центр керованого внутрішнього сегмента
* з axis (`x` або `y`)

---

# Чому в тебе зараз жовта стає крайовою

Бо зараз `waypoint` у тебе використовується як **реальна точка маршруту**, тобто вершина polyline.

А в draw.io логіка інша:

> жовта точка — це не вершина polyline, а **контролер сегмента**, який породжує 2 злами навколо себе.

Тобто:

* не `... -> waypoint -> ...`
* а `... -> bend-before -> segment-through-waypoint-center -> bend-after -> ...`

Саме тому на твоєму фото жовта точка опинилась у краю сегмента — бо система трактує її не як центр нового сегмента, а як вершину.

---

# Як має бути правильно

---

## Вихідна ламана без path point

Маєш 2 сегменти:

* горизонтальний
* вертикальний

І 2 сині точки:

* центр горизонтального
* центр вертикального

Це правильно.

---

## Додаєш path point на горизонтальний сегмент

Система не повинна “накладати жовту на синю”.

Вона має:

* взяти цей сегмент
* розбити його на **3 частини**
* утворити новий короткий сегмент
* жовту поставити **в центр цього нового сегмента**
* а сині handles перебудувати для нових сегментів

Тобто після додавання path point:

* стара синя точка того сегмента зникає
* з’являються **нові сині** для нових сегментів
* жовта живе окремо як `pathHandle`

Саме це видно в draw.io на твоїх скрінах.

---

# Тому правильний план виправлення такий

Я б робив це **не все одразу**, а в 3 кроки.

---

# Крок 1 — виправити сині точки

Це перший і обов’язковий крок.

## Треба домогтися:

* сині точки **ніколи не зберігаються в route**
* вони щоразу обчислюються з `geometry.points`
* вони завжди в центрі сегмента
* drag синьої точки:

  * не додає waypoint
  * не створює нову вершину
  * тільки змінює положення сегмента по осі

## Це означає:

поточний `moveElbowSegmentHandle()` має міняти не polyline напряму, а **модель сегмента**, наприклад через:

* `baseOffset` для базового elbow
* або через `segmentOffsets[]` для складніших випадків

На цьому кроці ми **не чіпаємо жовті точки**, поки сині не стануть правильними.

---

# Крок 2 — виправити модель жовтої точки

Коли сині вже нормальні, тоді:

## жовта точка повинна:

* зберігатись окремо як `pathHandle`
* мати:

  * `parentSegmentId`
  * `axis`
  * `offset/value`
* не бути вершиною polyline

## Після цього build geometry має:

* брати базову лінію
* якщо є `pathHandle` на сегменті —
  вставляти навколо нього 2 злами
* а жовту точку ставити в центр нового внутрішнього сегмента

Тоді вона буде поводитись як у draw.io.

---

# Крок 3 — перебудова синіх після додавання жовтої

Після того як додали path point:

* старий сегмент перестає існувати у попередньому вигляді
* значить стара синя точка на ньому теж не повинна існувати

Треба:

* заново побудувати список сегментів
* заново порахувати центри
* показати сині точки тільки для актуальних сегментів

---

# Найважливіше рішення зараз

## Не намагайся лікувати це дрібними патчами

бо основна проблема — не в `dragBoundFunc`, а в **моделі даних**.

Тобі треба чітко розвести:

### Сині точки

це **computed UI handles**, не data model

### Жовті точки

це **route controls**, але не vertex polyline

---

# Як я бачу правильну цільову модель

Для `orthogonal-elbow` приблизно так:

```js
route: {
  baseLayout: "hv" | "vh",
  baseOffset: { axis: "x" | "y", value: number } | null,

  pathHandles: [
    {
      id,
      segmentId,
      axis: "x" | "y",
      value: number
    }
  ]
}
```

А не:

```js
route: {
  waypoints: [{ x, y }]
}
```

для draw.io-подібної поведінки.

Бо `waypoint: {x,y}` занадто слабка модель для такого UX.

# Конкретно для поточного стану

На основі твоїх фото:

* синя точка зараз частково поводиться як “редактор вершини”
* жовта точка зараз поводиться як “vertex”
* а треба навпаки:

  * синя = segment mover
  * жовта = path segment controller

# Якщо коротко

Твій головний баг зараз такий:

> **жовта точка використовується як вершина маршруту, а має бути центром нового керованого сегмента**

І через це:

* вона перекриває синю
* вона стає крайовою
* синя й жовта починають конфліктувати
* перетягування не працює як у draw.io
