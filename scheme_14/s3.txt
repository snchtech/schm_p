Добре. Тоді **крок 2 — жовті точки**.

Тут головна ідея така:

> **жовта точка не повинна бути вершиною polyline**
> вона має бути **центром нового керованого відрізка**

Саме це зараз у тебе зламано.

---

# 1. Що має бути у draw.io-логіці

## Синя точка

* центр існуючого сегмента
* рухає сегмент
* не зберігається в route

## Жовта точка

* додається через “Додати точку шляху”
* не є кутом
* є **центром нового внутрішнього сегмента**
* при drag:

  * на горизонтальному сегменті рухається тільки по `y`
  * на вертикальному тільки по `x`
* після drag навколо неї мають утворитись **два злами**
* після перебудови сині точки з’являються на нових сегментах

---

# 2. Чому зараз у тебе погано

Бо зараз жовта точка, скоріш за все, зберігається так:

```js id="step2-01"
{ id, x, y, axis }
```

і далі прямо вставляється в polyline як вузол маршруту.

Тобто система сприймає її як:

```text
... -> waypoint -> ...
```

А треба сприймати як:

```text
... -> bend-before -> short-segment-centered-at-waypoint -> bend-after -> ...
```

Оце ключова різниця.

---

# 3. Нова роль жовтої точки

Я б пропонував для кроку 2 мислити її не як `waypoint`, а як:

## `pathHandle`

або можеш поки лишити назву `waypoint`, але по суті це має бути:

* не вершина
* а **контролер внутрішнього сегмента**

Тобто структура має бути приблизно така:

```js id="step2-02"
{
  id,
  segmentIndex,   // або stable id сегмента, якщо додаси пізніше
  axis: "x" | "y",
  x,
  y,
}
```

Але головне — **в geometry builder ця точка не вставляється напряму в polyline**.

---

# 4. Що має відбуватись при “Додати точку шляху”

Припустимо, є горизонтальний сегмент:

```text
A ---------------- B
        (синя)
```

Ти додаєш жовту точку.

Тоді не повинно бути так:

```text
A -------- Y -------- B
```

Бо тоді `Y` — вершина.

Має бути так:

```text
A -----┐
      └==== small segment ====┐
                              B
```

А жовта точка стоїть **в центрі цього нового small segment**.

Тобто:

* старий сегмент розбивається
* виникає новий короткий відрізок
* жовта = центр цього нового відрізка
* сині = центри вже нових сегментів

---

# 5. Мінімальна модель для кроку 2

Щоб не ламати все різко, я б радив таку модель у `route`:

```js id="step2-03"
route: {
  baseOffset: ...,
  waypoints: [
    {
      id,
      segmentIndex,
      axis: "y", // якщо точка додана на горизонтальний сегмент
      x,
      y,
    }
  ]
}
```

Але ще раз: `x,y` — це не vertex polyline, а **центр керованого внутрішнього сегмента**.

---

# 6. Як builder має трактувати жовту точку

Оце найголовніше в кроці 2.

## Якщо waypoint доданий на горизонтальний сегмент

і має `axis: "y"`:

то geometry builder повинен:

* взяти сегмент
* вставити дві нові точки зламу
* між ними зробити новий короткий горизонтальний сегмент на висоті `waypoint.y`
* жовта точка = центр цього короткого сегмента

## Якщо waypoint доданий на вертикальний сегмент

і має `axis: "x"`:

builder повинен:

* вставити дві нові точки зламу
* створити короткий вертикальний сегмент на `waypoint.x`
* жовта точка = центр цього сегмента

---

# 7. Чому жовта зараз накладається на синю

Бо ти додаєш її в ту саму “логічну позицію сегмента”, не перебудовуючи сам сегмент у нову структуру.

Тобто:

* стара синя точка лишається на старому сегменті
* жовта стає туди ж
* але нового внутрішнього сегмента фактично нема

Саме тому:

* вони перекриваються
* жовта стає крайовою
* drag працює дивно

---

# 8. Правильне правило після додавання жовтої

Після `add path point`:

## старий сегмент у старому вигляді більше не існує

Тому:

* стара синя точка на ньому зникає
* з’являються **нові сині** на нових сегментах
* жовта стоїть окремо в центрі свого нового сегмента

Тобто сині завжди обчислюються **після** перебудови geometry.

---

# 9. Що треба змінити в логіці drag жовтої точки

Зараз у тебе, судячи з опису, `moveConnectorWaypoint()` просто міняє `x/y` і rebuild не знає, як перетворити це в “новий короткий сегмент”.

Через це:

* сама жовта їздить
* а лінія не перебудовується правильно

## Правильна поведінка

При drag жовтої:

* змінюється її `x` або `y` по осі
* geometry builder перебудовує **новий внутрішній сегмент**
* жовта після rebuild знову стоїть у центрі цього сегмента

Тобто жовта теж **не повинна просто вільно їздити по старій polyline**.

---

# 10. Дуже важливе обмеження для MVP

Ти правильно підмітив:

> в draw.io на окремому відрізку зазвичай одна точка шляху

Для першого робочого варіанту я б теж так зробив:

## правило

* на одному сегменті може бути тільки **одна жовта точка**
* якщо сегмент уже має path point — не даємо додати ще одну

Це сильно спростить логіку.
