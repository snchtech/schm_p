В тебе вже зараз дуже хороша архітектура для такого функціоналу.
Ти фактично вже підготував основу через:

* `behavior`
* `pathType`
* `attached`
* централізований `canvasController`
* централізований Zustand store
* окрему модель connector-а 

Тому я б НЕ робив окремий "тип лінії".
Я б робив саме **behavior типу "link"**.

---

# 1. Головна ідея

Зараз у тебе є:

```js
behavior: {
   type: "default",
   link: null,
}
```



Оце і є правильне місце.

Тобто:

```js
behavior: {
   type: "link",
   link: {
      ...
   }
}
```

---

# 2. Що таке connector-link архітектурно

Я б трактував його так:

> "Цей connector не просто графічна лінія, а semantic navigation edge"

Тобто:

* він візуально такий самий connector
* але має metadata
* при click/doubleClick/context menu:

  * відкриває іншу схему
  * або фокусує елемент
  * або показує список схем

---

# 3. Найважливіше питання — як зберігати link

Оце ключова частина.

Ти написав:

> Один і той же id може бути на декількох схемах

І це абсолютно нормально.

Тому НЕ МОЖНА робити:

```js
targetSchemeId
```

бо це зламається.

---

# 4. Правильна архітектура — link by entity

Тобто link має бути не:

```js
schemeId
```

а:

```js
entityId
```

Наприклад:

```js
behavior: {
   type: "link",
   link: {
      entityType: "equipment",
      entityId: "pump_120",

      preferredSchemeId: null,

      openMode: "chooser"
   }
}
```

---

# 5. Чому це правильний підхід

Бо тоді:

## одна сутність:

```txt
pump_120
```

може бути:

* на overview схемі
* на детальній схемі
* на технологічній схемі
* на electrical схемі
* на аварійній схемі

---

# 6. Тоді navigation працює так

## connector click:

```txt
find schemes where entityId = pump_120
```

і:

* якщо 1 схема → відкриваємо
* якщо декілька → modal chooser

# 10. Яку metadata я б додав

Я б розширив:

```js

```

до:

```js
behavior: {
   type: "link",

   link: {
      entityType: "equipment",
      entityId: "pump_120",

      preferredSchemeId: null,

      openMode: "chooser",

      highlightTarget: true,

      label: "Насос 120"
   }
}
```

---

# 11. preferredSchemeId

Це дуже важлива штука.

Наприклад:

* є 10 схем
* але користувач хоче відкривати конкретну

Тоді:

```js
preferredSchemeId
```

як soft reference.

---

# 12. Як відкривати

При click:

```js
handleConnectorOpenLink(connector)
```

---

## алгоритм:

### 1

беремо:

```js
connector.behavior.link.entityId
```

### 2

робимо API:

```sql
SELECT scheme_id
FROM scheme_entities
WHERE entity_id = :entityId
```

---

### 3

if:

* 0 → повідомлення
* 1 → open
* many:

  * якщо preferredSchemeId → open it
  * else chooser modal

---

# 22. Головна порада

Твоя найбільша помилка зараз була б:

```txt
connector -> scheme
```

Правильна модель:

```txt
connector
   -> entity
      -> scheme(s)
         -> node
```

Саме так роблять великі SCADA/CAD/navigation systems.
